# DriveTest Agent 设计规格

## 1. 项目目标

DriveTest Agent 是一个面向智驾软件集成场景的 AI 测试生成原型。它解决“组件接口或需求发生变化后，人工编写回归测试耗时且容易遗漏测试规范”的问题。

项目需要在一周、每天 1–2 小时的投入下完成，并让 Python 基础水平的开发者能够理解和讲解全部核心代码。成功标准不是堆叠框架，而是构建一个可运行、可验证、可解释的 AI 工程闭环。

核心流程：

`需求/变更说明 → 检索测试规范 → 生成 pytest → 受限执行 → 分析结果 → 最多修正一次 → 输出报告`

## 2. 目标用户与演示场景

目标用户是负责智驾组件集成、回归测试和质量保障的开发或测试工程师。

标准演示输入：

> AEB 模块新增：当 TTC 小于等于 1.5 秒且相对速度为正时触发制动。

系统检索本地测试规范中的边界值、输入约束与测试命名规则，生成包含正常值、阈值、阈值两侧和异常输入的 pytest，用工具执行测试，并展示规范引用、执行结果和修正记录。

项目内置三个固定验收案例：

1. 信息完整的正常需求。
2. 容易遗漏等号或阈值两侧用例的边界需求。
3. 缺少关键参数、无法可靠生成测试的信息不足需求。

## 3. 范围

### 3.1 首版包含

- Streamlit 单页演示界面。
- 本地 Markdown 测试规范知识库。
- 轻量 RAG 检索，返回内容及来源。
- 单 Agent 编排循环。
- 基于 Pydantic 的结构化测试计划和结果。
- pytest 代码生成、受限执行和一次修正机会。
- 模拟 AEB Python 组件。
- 结构化测试报告。
- Fake LLM 驱动的自动化测试。
- GitHub Actions 持续集成配置。
- 架构说明、五分钟演示脚本和七天学习清单。

### 3.2 首版不包含

- 自动修改生产代码。
- 真实车辆数据、CAN 总线或真实量产系统接入。
- 用户系统、复杂权限控制和公网部署。
- 长期记忆、多个 Agent 或自主无限循环。
- 将 LangChain、LangGraph、AutoGen 作为核心依赖。
- MCP 服务端；MCP 仅作为后续演进方向。

## 4. 架构与组件

### 4.1 Streamlit 界面

负责收集需求文本和可选的组件说明，触发一次 Agent 运行，并展示：

- 检索到的规范片段与文件来源。
- 结构化测试计划。
- 生成的 pytest 代码。
- 首次执行和修正执行结果。
- 通过率、耗时、token 用量及最终状态。

界面不包含业务编排逻辑，只调用应用服务层。

### 4.2 文档知识库与检索器

知识库由 3–5 份短 Markdown 文档组成，分别描述：

- AEB 领域与输入约束。
- 边界值和异常值测试规范。
- pytest 命名与断言规范。
- 集成回归测试检查清单。

检索器将文档按标题和段落切分为带来源元数据的小片段，使用 scikit-learn 的 TF-IDF 与余弦相似度返回最多 3 个相关片段。最高相关度低于配置阈值时，系统必须标记低置信度，不能声称测试符合规范。为控制一周范围，首版不引入向量数据库或 Embedding API。

### 4.3 Agent 编排器

Agent 使用显式、有限状态流程，不依赖隐藏的框架循环。状态包含：

- 原始需求。
- 检索上下文。
- 测试计划。
- 当前测试代码。
- 执行结果。
- 修正次数。
- token 与耗时统计。

Agent 先请求模型输出符合 Pydantic Schema 的测试计划和代码，再调用 pytest 工具。首次失败时，将经过截断的错误信息交给模型修正一次；第二次仍失败则停止并生成失败报告。

单 Agent 是有意的工程取舍：当前任务步骤固定、上下文较小，多 Agent 会增加成本、延迟和调试难度。README 说明何时值得拆分为规划、执行和审查 Agent。

### 4.4 受限 pytest 工具

工具只允许在项目指定的临时测试目录中写入和执行生成的测试。它需要：

- 使用固定 Python 解释器运行 pytest。
- 设置执行超时。
- 限制可导入的项目模块。
- 截断 stdout 和 stderr。
- 返回退出码、通过数、失败数、耗时和错误摘要。
- 每次执行后清理临时文件。

这不是强安全沙箱。README 必须明确：原型不应执行不可信公网用户提交的代码；生产化需要容器、资源配额、网络隔离和更严格的系统调用限制。

### 4.5 模拟 AEB 组件

模拟组件保持小型、确定性和可测试，至少提供一个制动判断函数。输入包括 TTC、相对速度和传感器有效状态；输出为是否触发制动或明确的输入错误。

其目的不是模拟完整 AEB 算法，而是提供面试官可快速理解的智驾业务语境及边界条件。

### 4.6 模型网关

模型网关封装 OpenAI-compatible API，负责：

- 从环境变量读取模型名、基础 URL 和密钥。
- 设置请求超时。
- 记录 token 用量和延迟。
- 将模型响应解析为 Pydantic 对象。
- 对格式非法的响应重试一次。

API 密钥不得写入仓库。项目提供 `.env.example`。

## 5. 数据流

1. 用户通过 Streamlit 提交需求。
2. 应用服务验证输入长度和必填字段。
3. 检索器返回规范片段、来源及相关度。
4. Agent 将需求和受控数量的检索片段传给模型。
5. 模型返回结构化测试计划与 pytest 代码。
6. Pydantic 校验输出；失败时进行一次格式修复请求。
7. pytest 工具在临时目录执行代码并返回结构化结果。
8. 若首次执行失败，Agent 携带必要的失败摘要请求一次修正并重新执行。
9. 报告生成器汇总规范引用、测试场景、两次执行记录、指标和最终结论。
10. Streamlit 渲染报告；应用不自动修改被测组件。

## 6. 上下文与 Prompt 设计

系统 Prompt 明确角色、允许使用的上下文、输出 Schema 和停止条件。用户需求、检索内容和执行错误使用不同的标记区隔，降低指令混淆风险。

上下文管理遵循以下规则：

- 最多传入固定数量的高相关规范片段。
- 每个片段保留来源，不传入整份文档。
- 执行日志仅保留失败测试、异常类型和末尾堆栈摘要。
- 修正请求复用测试计划，但不重复无关页面内容。
- 最多一次格式重试和一次代码修正，不允许无限循环。

## 7. 错误处理

- 模型 API 不可用：显示明确错误并保留用户输入，不生成虚假结果。
- 检索无结果：标记低置信度，要求补充信息或说明测试缺少规范依据。
- 模型输出非法：Pydantic 校验失败后重试一次，仍失败则终止。
- pytest 超时：终止进程并返回超时状态。
- 输出过长：保留关键摘要并注明已截断。
- 首次测试失败：允许修正一次。
- 修正后仍失败：输出失败报告和根因摘要，不再重试。
- 需求信息不足：返回需要补充的字段，不生成看似完整的测试。

## 8. 测试与验收

### 8.1 单元测试

- Markdown 切分与来源元数据。
- TF-IDF 检索、Top-K 限制和低置信度判断。
- Pydantic 输出解析。
- 日志截断。
- 超时处理。
- Agent 的重试次数和停止条件。
- 报告指标计算。

### 8.2 集成测试

使用 Fake LLM 返回固定结构化响应，覆盖：

- 首次测试通过。
- 首次失败、修正后通过。
- 两次均失败并正确停止。
- 非法模型输出重试后失败。
- 检索无结果。

CI 不调用付费模型 API，保证测试可复现且无密钥也能运行。

### 8.3 演示验收

三个固定案例均可从 Streamlit 完成。系统需要展示：

- 测试生成成功与否。
- 首次通过率。
- 修正后通过率。
- 引用的规范来源。
- 总耗时和模型 token 用量。
- 失败时的明确原因。

## 9. 技术栈

- Python 3.11。
- Streamlit。
- pytest。
- Pydantic。
- OpenAI-compatible Python SDK。
- scikit-learn 的内存 TF-IDF 与余弦检索。
- Ruff 进行格式与静态检查。
- GitHub Actions 运行检查和测试。

依赖数量保持最少。首版直接实现 Agent 状态循环，以便学习和讲解；README 单独比较直接实现与 LangGraph 的取舍。

## 10. 交付物

- 可运行源码和配置示例。
- 模拟 AEB 组件及其手写基准测试。
- 本地测试规范知识库。
- Fake LLM 测试夹具。
- GitHub Actions 工作流。
- README：问题、架构、运行方式、演示步骤、设计取舍、局限与演进。
- Mermaid 架构图。
- 五分钟面试演示脚本。
- 七天学习清单，每天控制在 1–2 小时。
- 常见面试追问及答案要点，包括单 Agent 选择、RAG 价值、代码执行风险、评测方式、上下文管理和 MCP 演进。

## 11. 七天完成边界

- 第 1 天：理解场景、模拟 AEB 组件与 pytest。
- 第 2 天：知识库、切分和检索。
- 第 3 天：模型网关、Schema 和 Prompt。
- 第 4 天：pytest 工具和受限执行。
- 第 5 天：Agent 闭环和 Streamlit 页面。
- 第 6 天：Fake LLM 测试、CI 和错误路径。
- 第 7 天：README、演示脚本、面试问答与完整彩排。

若进度不足，按以下顺序降级：TF-IDF 检索改为词项重叠排序；减少界面样式；只保留两个演示案例。不得删除 Fake LLM 集成测试、执行超时、停止条件和错误报告，因为这些是工程可信度的核心。

## 12. 后续演进

- 将 pytest 执行器暴露为 MCP Tool，供其他 Agent 客户端调用。
- 当任务复杂度提高时，将单 Agent 拆为测试规划、执行和审查 Agent。
- 使用 LangGraph 持久化状态并支持人工审批节点。
- 接入真实 CI 平台的构建日志和变更信息。
- 使用容器隔离生成代码的执行环境。
- 建立离线评测集，比较不同 Prompt、模型和检索策略。
