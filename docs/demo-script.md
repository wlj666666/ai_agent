# 5 分钟面试演示脚本

本脚本可以照读。目标是在 5 分钟内讲清楚：系统做什么、怎么做、以及它诚实的边界在哪里。所有点击步骤对应 `streamlit run app.py` 启动后的真实页面。

## 准备工作（演示开始前完成，不计入 5 分钟）

```powershell
# 已完成 README 第 8 节的安装步骤，且 .env 中已填入可用的 OPENAI_API_KEY
streamlit run app.py
```

确认浏览器已打开页面，标题为「DriveTest Agent 演示」，下拉框「选择一个固定演示案例」中能看到三个案例。若无法联网或没有可用的 API Key，直接跳到本文末尾的**备用方案**，不要在台上临时排障。

## 时间分段

### 0:00–0:40 —— 开场：问题与目标

> 「智驾组件的需求或接口一旦变更，人工写回归测试很容易漏掉边界条件——比如等号方向、阈值两侧。这个项目做的是：输入一句需求变更说明，系统自动检索本地测试规范、生成 pytest 用例、在受限环境里跑一遍，失败了最多修正一次，最后给出一份可解释的报告，写清楚测试是基于哪份规范生成的、跑了几次、通过率多少。」

> 「这是一个学习/面试用原型，不是量产系统——这点我会在最后专门说清楚它的边界。」

### 0:40–1:10 —— 架构 30 秒版

> 「架构很直接：Streamlit 页面 → 业务装配层 → 一个单 Agent 编排器。Agent 先调用检索器在本地 4 份 Markdown 规范里做 TF-IDF 检索，再把需求和检索到的规范片段一起发给模型，模型返回结构化的测试计划和 pytest 代码，Pydantic 校验通过后在一个隔离临时目录里跑 pytest。第一次没全过，允许模型看到失败摘要后修正一次；两次都没过就停止，不会无限重试。」

（不需要真的打开架构图，口述即可；如果面试官感兴趣，可以打开 README 第 5 节的 Mermaid 图。）

### 1:10–2:20 —— 案例一：信息完整的正常需求

点击步骤：

1. 下拉框选择「案例一：信息完整的正常需求」。
2. 点击「填充该案例到需求输入框」按钮——需求文本和组件说明会自动填入两个文本框。
3. 点击「运行」按钮。

讲解要点（等待模型返回时口头讲）：

> 「案例一是一条信息完整的需求：TTC 小于等于 1.5 秒、相对速度为正、传感器有效时触发制动。系统会先展示检索到的规范来源和相关度，然后是模型生成的测试计划——几个用例名字和预期结果——再是生成的 pytest 代码，最后是执行记录和五个指标：首次通过率、修正后通过率、修正次数、总耗时、token 用量。」

运行结束后，依次指给面试官看：「检索到的测试规范」（展开一个来源看相关度和片段）、「测试计划」、「最终 pytest 代码」、「执行记录」、顶部五个指标卡片、以及「最终状态：成功」。

### 2:20–3:30 —— 案例二：容易遗漏的边界需求

点击步骤：

1. 下拉框切换到「案例二：TTC 等号与阈值两侧的边界需求」。
2. 点击「填充该案例到需求输入框」。
3. 点击「运行」。

讲解要点：

> 「案例二故意提高难度：需求里明确要求覆盖 TTC 等于 1.5 秒这个等号边界，以及 1.49 秒、1.51 秒两个阈值两侧的用例——这类等号方向和边界两侧的用例是人工写测试时最容易漏的。检索到的知识库片段里包含专门讲边界测试的规范文档，模型会依据这份规范生成对应用例。如果第一次生成的代码没有全部通过（比如漏了某个边界），系统会把失败摘要截断后再给模型一次修正机会，这里能看到『修正次数』变成 1，以及两次执行记录。」

如果案例二第一次就全部通过（说明模型这次生成得很完整），可以补一句：「这次模型第一次就覆盖全了，我们也内置了失败-修正路径，我可以用测试套件里的 Fake LLM 集成测试给你看这条路径的确定性验证」（引用 `tests/agent/test_orchestrator.py` 中 `TestDriveTestAgentCorrectionPath`）。

### 3:30–4:15 —— 案例三：信息不足的需求

点击步骤：

1. 下拉框切换到「案例三：缺少关键触发条件的信息不足需求」。
2. 点击「填充该案例到需求输入框」（注意这条需求和车机天气小组件有关，跟 AEB 知识库完全不相关）。
3. 点击「运行」。

讲解要点：

> 「案例三是故意设计的反例：这条需求跟本地知识库里的 AEB 规范完全不相关。检索器会发现最高相关度低于配置的阈值（默认 0.15），系统直接返回『信息不足』，不会调用模型编造一份看起来完整但没有依据的测试。这是我认为比『多生成几个用例』更重要的工程判断——诚实地说『不知道』，而不是假装知道。」

指给面试官看：最终状态显示「信息不足」，摘要文案说明是检索置信度不够，且**没有**生成任何测试代码或执行记录。

### 4:15–4:50 —— 边界与局限（主动说，不等被问）

> 「几点边界我想主动说清楚：第一，这里的 pytest 执行器不是强安全沙箱，只做了 AST 级别的导入/调用黑白名单加执行超时，绝不能拿来跑不可信的公网用户代码。第二，检索是 TF-IDF 词法匹配，不是语义检索，知识库也只有 4 份文档。第三，最多修正一次是有意的边界，不是遗漏。这些都写在 README 的『安全边界』和『已知局限』里。」

### 4:50–5:00 —— 收尾

> 「测试方面，全部自动化测试都不依赖真实 API Key，CI 里跑 `ruff check .` 和 `pytest -q` 就能验证，用的是 Fake LLM 打桩。这个项目下一步比较自然的演进是把 pytest 执行器封装成 MCP Tool，或者在任务复杂度提高后拆成规划/执行/审查多个 Agent，这些都在 README 最后一节写了权衡。」

## 失败时的备用方案

现场演示任何环节失败时，按以下优先级处理，**不要在台上排障超过 10 秒**：

1. **模型 API 超时/不可用/网络问题**：直接说明——「现在环境访问模型 API 不稳定，我用测试套件里的 Fake LLM 给你看同样的链路，它是完全确定性的，不依赖网络。」然后在终端运行：

   ```powershell
   pytest -q tests/agent/test_orchestrator.py -v
   ```

   逐个指出 `TestDriveTestAgentSuccessPath`（首次通过）、`TestDriveTestAgentCorrectionPath`（失败后修正通过）、`TestDriveTestAgentFailurePath`（两次失败后停止）、`TestDriveTestAgentInsufficientInfo`（低置信度直接返回信息不足）——这四个类分别对应案例演示里的四种结局，用代码和断言讲清楚，比等一个卡住的网络请求更可信。

2. **忘记配置 `OPENAI_API_KEY` 或 Key 已失效**：这本身也是一个可以讲的演示点——直接点击「运行」，页面会展示「未检测到 OPENAI_API_KEY 环境变量」的明确错误，而不是卡死或抛出堆栈。可以说：「这也是我特意测试过的一条路径——没配置密钥时系统会给出安全、清晰的错误，不会尝试悄悄发请求。」对应测试见 `tests/ui/test_service.py::TestMissingLlmConfigMessage` 和 `tests/ui/test_app_smoke.py::TestRunWithoutApiKey`。

3. **应用启动失败（`streamlit run app.py` 报错）**：说明本地环境问题，切换到直接阅读代码：打开 `src/drivetest_agent/agent/orchestrator.py` 的 `DriveTestAgent.run` 方法，逐行讲解状态机；或打开 `README.md` 第 5 节的架构图和第 6 节的数据流描述作为讲解载体。

4. **时间不够（面试官提前打断或时间被压缩）**：优先保留「案例三：信息不足」的演示或口述——这是最能体现工程判断而不是单纯堆功能的一个点，其次是收尾的边界/局限说明。案例一、二可以合并成一句话带过：「案例一二分别验证正常路径和边界/修正路径，逻辑和案例三类似，我可以之后补充演示或直接看测试代码。」
